# 方法论详解

## STEP 1: 反思结果

### 1.1 KPT

#### 起源
源自美國聖地牙哥大學的敏捷開发实践。

#### 核心結構

| 分類 | 簡稱 | 含義 | 用途 |
|------|------|------|------|
| Keep | K | 做得好，要保留 | 繼續做的事 |
| Problem | P | 有問題，需要改善 | 需要解決的痛點 |
| Try | T | 想要嘗試的新方法 | 實驗性改進 |

#### 使用步驟

**階段1：個人寫作 (5分鐘)**
1. 每個人獨立寫下KPT
2. 不限數量，想到什麼寫什麼
3. 不評價，只記錄

**階段2：分享討論 (15-25分鐘)**
1. 逐項分享
2. 分組討論相似議題
3. 找出共識點

**階段3：行動決定 (10分鐘)**
1. 選定優先處理的P項
2. 討論T項的可行性
3. 形成具體行動項目

#### 不同類型的KPT

**專案結束KPT**
- K: 專案中做得好的地方
- P: 專案中遇到的問題
- T: 下個專案想嘗試的事

**會議KPT**
- K: 會議運作良好的部分
- P: 會議運作有問題的部分
- T: 下次會議想改善的事

**階段KPT**
- K: 本階段成果
- P: 本階段問題
- T: 下階段計畫

#### 實用技巧

**專注於行動**
- KPT不是抱怨會議
- 每個P都希望有對應的T
- T應該具體可執行

**數量控制**
- 每個欄位3-5項最理想
- 過多會分散注意力
- 過少可能遺漏重點

**參與原則**
- 每個人都發言
- 不評價他人意見
- 記錄所有想法

### 1.2 YWT

#### 核心結構

| 分類 | 簡稱 | 含義 | 對象 |
|------|------|------|------|
| 讚 | Y | 讚賞做得好的地方 | 給他人 |
| 感 | W | 感謝他人的貢獻 | 給他人 |
| 想 | T | 想要改進或嘗試的事 | 給自己 |

#### 使用步驟

**準備**
1. 名字貼紙（每個參與者幾張）
2. 寫下給他人或自己的YWT
3. 不限人數，不限數量

**分享**
1. 輪流將貼紙貼在對應人員身上或貼板上
2. 對象可以口頭回應（可選）
3. 主持人記錄重點

**收斂**
1. 每個人總結收到的YWT
2. 找出共通性
3. 形成改進方向

#### 推薦比例

**Y:W:T = 3:2:1**

這個比例確保：
- 足夠的正面回饋（Y + W）
- 一點改進方向（T）

#### 不同場景的YWT

**專案結束**
- Y: 「你的分析很深入」
- W: 「謝謝你總是及時回應」
- T: 「下次我想更加主動參與」

**協作完成**
- Y: 「你的報告寫得很清楚」
- W: 「感謝你在這些方面的幫助」
- T: 「下個專案我想改善時間管理」

#### 實用技巧

**誠實原則**
- Y和W要真誠，不要客套
- 具體指出做得好的地方
- 提供可操作的感謝

**收斂T**
- T不是自我批評
- 是自主的改進決心
- 應該小而具體

**平衡給予**
- 平均分配Y和W給成員
- 避免某些人被忽略

### 1.3 PDCA 檢核表

#### 起源
由 Walter Shewhart 提出，經 W. Edwards Deming 推廣。

#### 四個階段

**P - Plan（計畫）**

| 檢核項 | 說明 |
|--------|------|
| 目標是否明確？ | 是否有清晰的目標值 |
| 計畫是否完整？ | 是否涵蓋所需步驟 |
| 資源是否充足？ | 人力、預算、時間是否就緒 |
| 風險是否評估？ | 是否預見可能的障礙 |

**D - Do（執行）**

| 檢核項 | 說明 |
|--------|------|
| 執行是否按計畫？ | 是否有重大偏差 |
| 進度是否符合預期？ | 是否按時完成 |
| 問題是否即時解決？ | 是否記錄並處理問題 |
| 資訊是否即時更新？ | 是否同步進度 |

**C - Check（檢查）**

| 檢核項 | 說明 |
|--------|------|
| 數據是否收集？ | 是否有客觀依據 |
| 結果是否與目標對比？ | 是否比較目標值 |
| 差異是否分析？ | 是否找出差距原因 |
| 原因是否深入？ | 是否找到根本原因 |

**A - Act（處理）**

| 檢核項 | 說明 |
|--------|------|
| 成功是否標準化？ | 好的做法是否制度化 |
| 失敗是否改善？ | 問題是否有改善方案 |
| 課題是否設定？ | 下一個PDCA的主題是否明確 |
| 責任是否分配？ | 誰負責後續行動 |

#### PDCA 循環圖示

```
    →→→→→→→
   ˏ     ˎ ˎ
 P → D ˎ
 A   →← C
 ˎ   ←← ˎ
 ←←← ←←
```

#### 應用範例

**場景：提升月報準時提交率**

| 階段 | 行動 | 檢核結果 |
|------|------|---------|
| P | 設定目標：下個月準時提交率90% | 目標明確 ✓ |
| D | 設定提醒機制、提前準備 | 執行按計畫 ✓ |
| C | 對比結果：實際達成95% | 超越目標 ✓ |
| A | 標準化提醒機制，推廣至其他部門 | 持續改進 ✓ |

## STEP 2: 業務狀態視覺化

### 2.1 業務盤點表

#### 結構與用途

| 欄位 | 含義 | 分析用途 |
|------|------|---------|
| 業務項目 | 業務名稱 | 識別 |
| 頻率 | 執行頻率 | 工作/週/月/季 |
| 負責人 | 主辦人員 | 權責 |
| 時間 | 執行時間 | 找出耗時項 |
| 重要度 | 隱性或顯性需求 | 確定重要性 |
| 問題點 | 遇到的困難 | 改善點 |
| 優先順序 | 改善優先級 | 排序 |

#### 分析方法

**找出高頻高耗時業務**
```markdown
業務盤點表分析

高頻率、高耗時（優先改善1）：
- 週報（每週，60分鐘）
- 會議整理（每週，40分鐘）

中頻率、高耗時（優先改善2）：
- 月報（每月，120分鐘）
年報（每年，480分鐘）

低頻率、低耗時（暫時不優先）：
- 季度總結（每季，30分鐘）
```

**價值分析**
- 對比重要度與現狀
- 找出「重要但做得不好」的業務
- 找出「不重要但耗時」的業務

#### 使用建議

1. **由全體成員一起填寫**
2. **誠實記錄實際時間**
3. **定期更新（每季）**

### 2.2 業務流程圖

#### 基本符號

```
  (開始)
   │
   ↓
┌────────┐
│ 步驟A  │
└───┬────┘
    │ 然後
    ↓
┌────────┐
│ 步驟B  │
└───┬────┘
    │ 決策
    ↓
   ◇↓◇
   │ 檢查
   ├── 是 ─→◇→(結束)
   │ 否
   ↓
┌────────┐
│ 步驟C  │
└────────┘
```

#### 分析要點

**找出多餘步驟**
- 是否有重複檢查？
- 是否有不必要的簽核？
- 是否有資訊重複輸入？

**找出等待時間點**
- 檔審批等待
- 上下串聯等待
- 約定他人時間等待

**找出資訊不連接**
- 線上與線下混亂
- 不同系統資料不相同步

#### 改進方法

1. 消除多餘步驟
2. 合併相似步驟
3. 並行可同步的工作
4. 自動化可自動的工作

### 2.3 PERT圖

#### 節點與關係

**節點**：代表任務
**有向邊**：代表依賴關係

```
A ──→ C ──→ E
│      │
↓      ↓
B ──→ D ──→ F

例子：
A和B都完成後，才能做C和D
C和D都完成後，才能做E和F
```

#### 關鍵路徑

**定義**：最長時間的任務鏈

**舉例**：

```
任務   時間   先置

A      3天    -
B      4天    -
C      2天    A
D      5天    A, B
E      3天    B

路徑分析：
路徑1 (A-C): 3 + 2 = 5天
路徑2 (A-D): 3 + 5 = 8天  ← 關鍵路徑
路徑3 (B-D): 4 + 5 = 9天  ← 關鍵路徑
路徑4 (B-E): 4 + 3 = 7天

最短總時間: 9天
關鍵路徑: B → D
```

**應用場景**
- 專案時程優化
- 識別瓶頸任務
- 資源分配決策

### 2.4 RACI 矩陣

#### 角色定義

| 角色 | 英文 | 中文 | 職責 |
|------|------|------|------|
| R | Responsible | 執行者 | 領取任務並執行 |
| A | Accountable | 負責人 | 最終負責職位/人 |
| C | Consulted | 諮詢者 | 提供意見的人 |
| I | Informed | 知情者 | 需要被通知的人 |

#### 矩陣範例

**場景：新產品推出**

| 任務 | 產品經理 | 工程師 | 設計師 | 行銷 | 總監 |
|------|---------|--------|--------|------|------|
| 產品定義 | A | R | C | C | I |
| 設計實作 | R | I | A | I | I |
| 開發實作 | R | A | C | I | I |
| 行銷規劃 | R | I | C | A | I |
| 預算批準 | R | I | I | I | A |

#### 重要原則

**一個任務一個A**
- 唯一負責人，避免責任模糊
- 追蹤時有明確聯絡對象
- 決策時有明確決定者

**可以有多個R**
- 需要明確分工
- 避免工作重疊

**C與I的區別**
- C需要主動參與，提供意見
- I只是被通知，不必參與

## STEP 3: 思考改善策略

### 3.1 勉強．不均．浪費 (Muri, Mura, Muda)

#### 原始含義

來自豐田生產方式（Toyota Production System）

| 日文 | 英文 | 中文 | 含義 |
|------|------|------|------|
| 無理 | Muri | 勉強/過載 | 超出能力範圍 |
| 斑 | Mura | 不均 | 工作負荷不穩定 |
| 無駄 | Muda | 浪費 | 不創造價值的活動 |

#### Muri（勉強）

**現象**：
- 超時工作頻繁
- 過度承諾
- 設備超載
- 員工疲勞

**分析問句**：
- 「誰一直在勉強工作？」
- 「哪個環节一直在超負荷？」
- 「什麼工作持續超出預算時間？」

**改善方向**：
- 增加人力或設備
- 調整工作排程
- 分散負荷
- 降低要求

#### Mura（不均）

**現象**：
- 今天忙明天閒
- 忽忙忽閒的工作模式
- 季節性高峰
- 資料更新時間不固定

**分析問句**：
- 「哪個時間段的工作負荷最不平均？」
- 「哪個人的工作波動最大？」
- 「什麼作業不定期？」

**改善方向**：
- 生產平準化
- 庫存管理
- 固定工作排程
- 自動化調度

#### Muda（浪費）

**八大浪費**：

| 類型 | 說明 | 範例 |
|------|------|------|
| 過量生產 | 生產超過需求 | 提前準備多餘報告 |
| 等待 | 上游作業完再可做 | 等待簽核 |
| 不必要輸送 | 不必要的移動 | 檔案線上傳下載 |
| 過多加工 | 加工超過需求 | 過細化的格式 |
| 存貨 | 不必要庫存 | 儲存多餘資料 |
| 不必要動作 | 不必要的人員動作 | 複雜的資料輸入 |
| 缺陷 | 重做或修正 | 錯誤需要重新來 |
| 未利用人才 | 未使用的能力 | 專才做雜事 |

**分析問句**：
- 「哪個步驟不創造價值？」
- 「哪個步驟可以直接刪除？」
- 「什麼作業可以合併？」

**改善方向**：
- 削除多餘步驟
- 合併相似作業
- 自動化機械性工作
- 簡化複雜程序

### 3.2 ECRS

#### 四個改進方向

| 方向 | 英文 | 中文 | 優先級 |
|------|------|------|--------|
| E | Eliminate | 削除 | 1（最優先） |
| C | Combine | 結合 | 2 |
| R | Rearrange | 重排 | 3 |
| S | Simplify | 簡化 | 4 |

#### 應用範例

**場景：報告審核流程**

**原流程（共5天）**：
1. 撰寫報告（1天）
2. 主管初審（0.5天）
3. 修改後副總審核（0.5天）
4. 再修改後總經理審核（0.5天）
5. 終版發布（0.5天）

**ECRS改善策略**：

| 方向 | 改善動作 | 結果 |
|------|---------|------|
| E | 削除：副總審核步驟，總經理可一次審核 | 節省0.5天 |
| C | 結合：主管與副總同時審核 | 節省0.5天 |
| R | 重排：修改後不再需要總經理額外審核 | 節省0.5天 |
| S | 簡化：改用線上審核系統 | 節省0.5天 |

**改善後流程（共2.5天）**：
1. 撰寫報告（1天）
2. 主管與副總同步審核（0.5天）
3. 總經理審核並發布（1天）

**效果**：
- 時間縮短：5天 → 2.5天（節省50%）
- 流程簡化：5步驟 → 3步驟
- 效率提升：減少等待時間

#### ECRS實施建議

**E - Eliminate**（最優先）
- 先問：「這個步驟能不能刪除？」
- 削除是最省力的改善
- 削除多餘簽核步驟

**C - Combine**
- 合併相似或相似步驟
- 讓相關人員同時參與
- 減少上傳下載次數

**R - Rearrange**
- 重新排列順序
- 讓工作並行化
- 減少前置等待

**S - Simplify**
- 簡化每個步驟
- 自動化機器工作
- 簡化表單與格式

- 目標：提升滿意度

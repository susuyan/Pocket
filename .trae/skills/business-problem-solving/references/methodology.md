# 方法论详解

## 四步框架概述

本框架提供系统化的商业问题解决方法，包含四个核心步骤：

```
差距分析 → 6W2H检视 → 5Why分析 → 可控性分析
```

---

## 步骤1: 差距分析

### 核心要素

**As is (现状)**
- 客观描述当前的情况
- 使用数据、事实、可观察的表现
- 避免主观判断和推测

**To be (理想)**
- 明确期望达到的目标
- 定义成功标准和状态
- 可量化的结果/指标

**Gap (落差)**
- 现状与理想之间的具体差距
- 可直接观察的差距表现
- 形成问题的直接证据

### 示例

问题：项目延期

| 要素 | 内容 |
|---|---|
| As is | 项目原计划3月完成，现已6个月，仅完成60% |
| To be | 3个月内完成项目全部功能 |
| Gap | 延期3个月，剩余40%工作量未完成 |

---

## 步骤2: 6W2H检视

### 八个维度详解

| 维度 | 问题 | 回答要点 |
|---|---|---|
| Who | 谁是相关者？谁受影响？ | 利益相关者、责任人、影响范围 |
| What | 发生了什么问题？ | 问题的具体表现和现象 |
| When | 问题何时发生？持续多久？ | 时间点、频率、持续时间 |
| Where | 问题发生在哪里？ | 地点/系统/部门/流程 |
| Why | 为什么这是个问题？ | 问题的重要性判断 |
| Which | 涉及哪些对象或选择？ | 具体对象、选项、变体 |
| How | 问题是如何发生的？ | 触发条件、发生路径 |
| How Much | 程度如何？损失多少？成本多少？ | 量化指标、影响程度 |

### 示例

延续项目延期案例：

| 维度 | 回答 |
|---|---|
| Who | 开发团队、产品经理、客户 |
| What | 项目交付延期3个月 |
| When | 从第4个月开始出现问题，持续3个月 |
| Where | 核心功能开发模块 |
| Why | 客户业务无法按计划上线，造成损失 |
| Which | 影响3个核心功能模块 |
| How | 需求变更频繁，返工率高 |
| How Much | 延期成本约50万，客户满意度下降 |

---

## 步骤3: 5 Why分析

### 方法

通过连续追问「为什么」，揭示问题的层层因果关系。

### 示例

项目延期案例的5Why：

```
问题: 项目延期3个月
  ↓ Why?
原因1: 需求变更频繁
  ↓ Why?
原因2: 需求未在开发前确定
  ↓ Why?
原因3: 产品与技术对需求理解不一致
  ↓ Why?
原因4: 缺乏正式的需求评审流程
  ↓ Why?
原因5 (根本原因): 项目启动时未建立需求治理机制
```

### 停止判断

- 找到可控的根本原因
- 原因链无法继续深入
- 达到系统/组织边界

---

## 步骤4: 可控性分析

### 分析方法

对于每个识别出的原因，判断可控性并制定对策。

### 示例

| 原因 | 可控性 | 对策 |
|---|---|---|
| 需求变更频繁 | 不可控制 | 建立需求变更评估机制 |
| 需求未在开发前确定 | 可控制 | 冻结开发前需求 |
| 产品与技术对需求理解不一致 | 可控制 | 增加需求评审会议 |
| 缺乏正式的需求评审流程 | 可控制 | 制定需求评审标准流程 |
| 未建立需求治理机制 | 可控制 | 设立项目治理委员会 |

### 对策类型

**可控制因素** → 具体行动计划
- 明确责任人
- 设定时间节点
- 定义成功标准

**不可控制因素** → 应对策略
- 监控机制
- 规避措施
- 缓解方案

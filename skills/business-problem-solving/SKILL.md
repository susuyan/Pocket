# 商业问题解决

## 任务目标
运用四步方法论帮助使用者系统化分析商业问题，将抽象问题具象化、可操作化。

## 触发方式

### 主动触发
当用户提出以下类型的需求时触发：
- "帮我分析这个问题"
- "不知道如何解决"
- "这个情况很复杂"
- 需要结构性分析的复杂场景

### 命令触发
```bash
/problem-solve [问题描述]
```

## 操作步骤

### 1. 差距分析 (Gap Analysis)

目标：明确问题主题，将理想与现状的落差可视化。

三个要素：

**As is (现状)**
- 客观描述当前情况
- 使用数据、事实、可观察的表现
- 避免主观判断

**To be (理想)**
- 明确期望达到的目标
- 定义成功标准
- 可量化的结果/指标

**Gap (落差)**
- 现状与理想之间的具体差距
- 可直接观察的差距表现
- 形成问题的直接证据

### 2. 6W2H 检视

目标：通过八个疑问词全面检视问题。

| 维度 | 问题 | 回答要点 |
|---|---|---|
| Who | 谁相关？谁受影响？ | 利益相关者、责任人、影响范围 |
| What | 什么问题？ | 问题的具体表现 |
| When | 何时发生？持续多久？ | 时间点、频率、持续时间 |
| Where | 在哪里发生？ | 地点/系统/部门/流程 |
| Why | 为什么是问题？ | 初步原因判断 |
| Which | 涉及哪些对象？ | 具体对象、选项、变体 |
| How | 如何发生？ | 触发条件、发生路径 |
| How Much | 程度如何？损失多少？ | 量化指标、影响程度 |

### 3. 5 Why 原因分析

目标：深究问题产生的根本原因。

方法：连续追问「为什么」

```
问题发生
  ↓ Why?
原因1
  ↓ Why?
原因2
  ↓ Why?
原因3
  ↓ Why?
原因4
  ↓ Why?
原因5 (根本原因)
```

停止条件：
- 找到可控的根本原因
- 原因链无法继续深入
- 达到系统/组织边界

### 4. 可控性分析

目标：掌握己方有能力改变的事物，聚焦资源。

对于每个识别出的原因：

**可控制因素**
- 有权/有能力改变
- 资源覆盖范围内
- 可以采取行动
→ 制定具体行动/对策

**不可控制因素**
- 外部因素或超出权限
- 需要协调或等待
→ 制定应对策略（监控、规避、缓解）

## 输出格式

完成分析后输出结构化报告，包含四个部分的完整分析结果。

报告格式详见 [references/report-format.md](references/report-format.md)

## 关键原则

- 客观性：仅描述事实，不作评价
- 数据驱动：可量化、可验证
- 边界明确：避免范围无限扩大
- 全面性：8个维度都应回答
- 聚焦根本：而非表面症状
- 聚焦可控：把精力放在能影响的范围内

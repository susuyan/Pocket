# 改善业务

## 任务目标
運用三步驟方法论幫助使用者系統化改善現有業務，從反思現狀到可視化分析，再到策略制定，形成完整的業務改進方案。

## 触发方式

### 主动触发
当用户提出以下类型的需求时触发：
- "改善业务流程"
- "提升工作效率"
- "业务有问题需要改進"
- "流程需要優化"
- 系統化業務診斷

### 命令触发
```bash
/improve [業務主題]
```

## 操作步骤

### STEP 1: 反思结果

目标：客觀檢視業務表現，識別問題與機會。

#### 1.1 KPT

核心：三欄式反思工具，專注「保持、問題、嘗試」。

| 分類 | 簡稱 | 內容 |
|------|------|------|
| Keep | K | 做得好，要保留的 |
| Problem | P | 有問題，需要改善的 |
| Try | T | 想要嘗試的新方法 |

**應用方式**：
1. 開會前，每個人以5分鐘寫下KPT
2. 逐項分享討論
3. 找出共識點，形成行動項目

**應用場景**：專案結束、會期結束、定期審查

#### 1.2 YWT

核心：以「讚、感、想」三個角度反思。

| 分類 | 簡稱 | 含義 |
|------|------|------|
| 讚 | Y | 讚賞他人/團隊做得好的地方 |
| 感 | W | 感恩對他人的貢獻 |
| 想 | T | 下次想改進或嘗試的事 |

**應用場景**：協作後、專案完成後、定期復盤

**使用提示**：
- Y和W回應給他人，T給自己
- 建議Y:W:T比例為 3:2:1

#### 1.3 PDCA 檢核表

核心：計畫-執行-檢查-處理的循環改進方法。

**四個階段**：

| 階段 | 英文 | 中文 | 目標 |
|------|------|------|------|
| P | Plan | 計畫 | 設定目標、計畫行動 |
| D | Do | 執行 | 執行計畫 |
| C | Check | 檢查 | 比較結果與目標 |
| A | Act | 處理 | 採取行動（標準化或調整） |

**檢核表範例**：

| 檢核項 | 說明 | 結果 |
|--------|------|------|
| 目標是否明確？ | 有無明確目標值 | ☐ 是 ☐ 否 |
| 執行是否按計畫？ | 偏差分析 | ☐ 是 ☐ 否 |
| 數據是否收集？ | 檢查依據 | ☐ 是 ☐ 否 |
| 原因是否分析？ | 問題根源 | ☐ 是 ☐ 否 |
| 改善是否規劃？ | 下一步行動 | ☐ 是 ☐ 否 |

**應用場景**：持續改進、品質管理

---

### STEP 2: 將業務狀態視覺化

目标：將抽象業務轉化為可視化的分析資料。

#### 2.1 業務盤點表

核心：記錄所有業務項目的狀態。

| 業務項目 | 頻率 | 負責人 | 時間 | 重要度 | 問題點 | 優先順序 |
|---------|------|--------|------|--------|--------|----------|
| 日報 | 每日 | 張三 | 30分 | 高 | 無 | - |
| 週會 | 每週 | 李四 | 60分 | 中 | 過長 | 2 |
| 月報 | 每月 | 王五 | 120分 | 高 | 自動化不足 | 1 |

**分析用途**：
- 識別高頻率業務（改善需求高）
- 找出耗時業務（優先改善）
- 評估重要度與現狀的差距

#### 2.2 業務流程圖

核心：流程步驟可視化，識別瓶頸。

**基本符號**：

| 符號 | 含義 |
|------|------|
| 橢圓 | 開始/結束 |
| 矩形 | 處理步驟 |
| 菱形 | 決策點 |
| 箭頭 | 順序方向 |

**分析要點**：
- 找出「多餘步驟」
- 找出「等待時間」點
- 找出「重複檢查」

#### 2.3 PERT圖

核心：專案關鍵路徑分析，用於��程規劃。

**節點關係**：
- A → B：A完成後B可開始
- A ║ B：A與B同時進行

**關鍵路徑**：
- 最長時間的任務鏈
- 決定專案最短完成時間

**應用場景**：專案時程優化、瓶頸識別

#### 2.4 RACI 矩陣

核心：明確每個任務的角色與責任。

| 字母 | 英文 | 中文 | 說明 |
|------|------|------|------|
| R | Responsible | 執行者 | 實際執行者 |
| A | Accountable | 負責人 | 最終負責者 |
| C | Consulted | 諮詢者 | 需要諮詢的人 |
| I | Informed | 知情者 | 需要被通知的人 |

**矩陣範例**：

| 任務 | 張三 | 李四 | 王五 | 陳六 |
|------|------|------|------|------|
| 方案設計 | R | C | C | I |
| 批准簽核 | I | A | C | I |
| 執行推動 | R | I | C | I |

**重要原則**：
- 每個任務只有一個 A（負責人）
- R 可能有多個，但要明確分工

### STEP 3: 思考改善策略

目标：基於分析結果，制定具體改善措施。

#### 3.1 勉強．過剩．不均 (Muri, Mura, Muda)

來自豐田生產方式的三大浪費類型：

**Muri（勉強）**：超出能力範圍
- 現象：超時工作、過度承諾、設備超載
- 改善：調整工作負荷、增加資源、優化調度

**Mura（不均）**：工作負荷不穩定
- 現象：忽忙忽閒、工作量波動
- 改善：平衡生產、庫存管理、生產平準化

**Muda（浪費/過剩）**：不創造價值的活動
- 現象：多餘步驟、等待、移動、過量生產
- 改善：消除浪費、簡化流程

**分析方式**：
1. 列出所有工作中可能的勉強、不均、浪費
2. 評估影響程度與改善可行性
3. 制定改善優先清單

#### 3.2 ECRS

核心：改進工作的四個方向。

| 字母 | 英文 | 中文 | 說明 |
|------|------|------|------|
| E | Eliminate | 削除 | 消除不必要的工作步驟 |
| C | Combine | 結合 | 合併相似或相關步驟 |
| R | Rearrange | 重排 | 調整步驟順序以提效 |
| S | Simplify | 簡化 | 簡化每個步驟 |

**應用步驟**：
1. 先考慮 E（削除最省力）
2. 再考慮 C（結合步驟）
3. 接著 R（重新排列）
4. 最後 S（簡化細節）

**範例**：

| 原流程 | ECRS改善 |
|--------|----------|
| 審核 → 主管簽核 → 副總簽核 → 總經理簽核 | 削除：同級簽核合併為一關 |
| 填寫表單 → 列印 → 簽名 → 掃描 | 簡化：改用數位簽名 |
| A做，再B做 | 重排：改為A與B同步進行 |

#### 3.3 業務改善提案表

核心：標準化改善提案格式。

**表單結構**：

| 欄位 | 內容 |
|------|------|
| 標題 | 改善提案的名稱 |
| 現況描述 | 目前的問題與影響 |
| 改善方案 | 具體改善措施 |
| 改善效果 | 預期的好處（量化） |
| 實施成本 | 預估成本與資源 |
| 實施時程 | 計畫執行時間 |
| 實施風險 | 可能遇到的困難 |
| 關鍵成功因素 | 確保成功的要點 |

**應用場景**：員工提案制度、專案改善、流程優化

## 籌畫會議時應留意的重點

### 會議前

**準備事項**：
- [ ] 確定會議目標
- [ ] 準備相關資料（數據、報告）
- [ ] 預先發送議程
- [ ] 確認參與者與角色

**議程設計**：
- 明確時間分配
- 列出重點議題
- 預留討論時間

### 會議中

**主持人職責**：
- 按時開始與結束
- 控制討論方向
- 鼓勵所有成員發言
- 記錄關鍵決議
- 明確行動項目（負責人、截止日）

**參與原則**：
- 預習資料
- 按時出席
- 積極發言
- 傾聽他人
- 聚焦議題

### 會議後

**後續動作**：
- 發送會議紀錄（24小時內）
- 追蹤行動項目執行
- 定期檢查進度
- 評估改善效果

**會議紀錄要素**：
- 會議日期、時間、地點
- 參與名單
- 討論重點
- 決議事項
- 行動清單（行動/負責人/截止日）
- 下次會議時間

## 完整工作流程

```
開始
 ↓
STEP 1: 反思結果
 ├─ KPT (Keep, Problem, Try)
 ├─ YWT (讚、感、想)
 └─ PDCA 檢核表
 ↓
識別問題與機會
 ↓
STEP 2: 業務狀態視覺化
 ├─ 業務盤點表
 ├─ 業務流程圖
 ├─ PERT圖
 └─ RACI 矩陣
 ↓
分析業務現狀
 ↓
STEP 3: 思考改善策略
 ├─ 勉強．過剩．不均分析
 ├─ ECRS改善
 └─ 業務改善提案表
 ↓
形成改善方案
 ↓
籌畫與執行改善會議
 ├─ 會議前準備
 ├─ 會議中執行
 └─ 會議後追蹤
 ↓
生成改善報告
 ↓
結束
```

## 关键原则

- **客觀性**：基於數據而非感覺
- **可視化**：流程問題一目了然
- **參與性**：全體成員參與改善
- **持續性**：PDCA循環，持續改進
- **可測量**：改善效果需可衡量
- **可落地**：改善方案需具體可行

完成分析後輸出結構化報告。
